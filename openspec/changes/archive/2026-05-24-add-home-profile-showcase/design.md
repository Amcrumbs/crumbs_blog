## Context

项目仍然遵循内容驱动原则，因此不引入数据库或 CMS，而是在现有私密解锁基础上补一个轻量编辑路径。用户已经明确要求这块内容保持克制黑白、偏开发者气质、避免求职简历感，并且需要中英文都可稳定展示。

## Goals / Non-Goals

**Goals:**

- 在首页欢迎栏下方增加一个公开的个人展示区，让访客快速认识站点主人并理解站点用途。
- 将展示内容抽成独立本地 JSON 文件，统一作为首页展示和私密区编辑的数据源。
- 在私密区新增一个小型内容面板，用于编辑个人展示区内容。
- 为中文和英文内容建立稳定的字段结构和布局安全区，避免长短差异破坏排版。
- 保持现有私密解锁方式、公开内容目录和站点黑白视觉方向不变。

**Non-Goals:**

- 不实现数据库、后台管理系统或通用 CMS。
- 不为首页展示区引入富文本编辑器、图片上传系统或媒体管理能力。
- 不在本次变更中实现插画素材本身，只提供插画占位字段和展示位。
- 不修改 notes、logs、tools、links、guestbook 的内容读取结构。
- 不引入新的身份系统、角色权限或多用户编辑能力。

## Decisions

### 1. 使用单独 JSON 文件作为展示区内容源

个人展示区内容将放在单独的本地 JSON 文件中，而不是塞进组件常量、字典文件或 Markdown frontmatter。

理由：

- 私密区表单编辑最适合映射到结构化 JSON。
- 这块内容同时服务公开展示和私密编辑，单文件更容易维护。
- 和现有内容驱动项目方向一致，但不会污染文章内容目录。

备选方案：

- 写进 `src/lib/i18n.ts`：不适合网页编辑回写，且会让字典承担结构化内容管理职责。
- 写成 Markdown/frontmatter：适合长文，不适合这种字段型个人资料和链接数组。
- 写进数据库：超出当前项目范围。

### 2. 将能力拆成“公开展示”与“私密编辑”两个层次

首页公开展示和私密区编辑面板属于同一业务域，但职责不同，因此在 spec 中拆为两个 capability。

理由：

- 首页需要定义展示结构、双语呈现和安全区约束。
- 私密区需要定义访问前提、表单字段映射和保存行为。
- 分开后更容易测试，也避免未来扩展时一个 spec 过于臃肿。

备选方案：

- 合并为一个 capability：可行，但会把公开行为、编辑行为和存储规则混在一起，后续维护不清晰。

### 3. 私密区编辑面板保持轻量结构化表单，而非 CMS 式后台

编辑界面采用小型内容面板，按“身份信息”“重点项目”“长期方向与兴趣”“社交链接”分组，而不是引入复杂的后台导航或所见即所得编辑器。

理由：

- 用户明确希望是“小型内容面板”。
- 这块内容字段有限，结构稳定，不值得引入重后台心智。
- 更符合当前项目轻量、直接、可维护的原则。

备选方案：

- 通用 CMS 式管理页：功能更强，但范围过大。
- 只改文件不做网页编辑：不满足用户要求。

### 4. 双语内容使用同一字段结构，但允许分别润色

JSON 结构中中英文使用同一字段键位，例如 `tagline.zh` / `tagline.en`、`project.summary.zh` / `project.summary.en`，但文案内容不要求逐字直译。

理由：

- 可以保证组件结构、字段校验和表单映射一致。
- 文案润色空间更大，不会因为强行直译影响英文自然度。

备选方案：

- 只存一种语言并运行时翻译：不符合内容可控性要求。
- 拆成两份完全独立结构：维护成本高且容易字段漂移。

### 5. 明确首页展示区的布局安全区

展示区应预设可控的长度和数量边界，尤其针对双语差异：

- 一句话介绍控制在短段落级别
- 工作方式技能控制为有限标签列表
- 社交链接控制为少量文字入口
- 重点项目固定 1 个
- 正在做的事控制为 2 到 4 条
- 兴趣方向使用短标签

理由：

- 用户已经明确希望“规范好安全区”。
- 这样可以降低中英文长度差异导致的卡片塌陷和换行失控风险。

备选方案：

- 不定义安全区：实现快，但后续文案一增长就容易破版。

## Risks / Trade-offs

- JSON 文件由网页直接回写，若字段过宽或结构漂移，容易造成保存逻辑脆弱 -> 通过固定字段结构和轻量校验降低风险。
- 私密区新增编辑能力会让该区域从“只读浏览”变成“可修改内容”的入口 -> 继续复用现有私密解锁前提，不额外暴露公开写接口。
- 双语文案长度不一致可能破坏卡片平衡 -> 通过字段数量和文案长度安全区约束控制。
- 重点项目当前先使用占位内容，后续真实项目替换时可能触发文案长度变化 -> 组件应为一句话描述预留弹性高度。

## Migration Plan

1. 新增个人展示区 JSON 文件，并写入当前确认过的初始双语内容。
2. 新增读取该 JSON 的内容模块，并在首页展示区中消费。
3. 在私密区加入展示区编辑面板，加载并保存同一份 JSON。
4. 验证未解锁用户无法看到编辑界面，已解锁用户可编辑并刷新首页后看到更新。
5. 运行类型检查、lint 和必要的构建验证。

本次没有数据库迁移。若需回滚，只需移除新增 JSON 文件、编辑逻辑和首页展示区组件，并恢复私密区原有结构。
