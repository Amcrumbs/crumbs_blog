# Agents Guide

## 项目目标

这是一个基于 Next.js、TypeScript 和内容文件驱动的个人博客 / 个人知识空间项目。项目的核心目标是：

- 稳定展示公开文章、日志、工具、链接、留言等内容。
- 支持中英文内容组织，保持内容结构清晰、可维护。
- 为个人长期写作、记录、整理和扩展功能提供简洁可靠的代码基础。
- 优先保证阅读体验、内容可维护性和实现清晰度，不追求不必要的复杂架构。

## 开发原则

- 模块化、组件化：按职责拆分页面、组件、数据读取、业务逻辑和工具函数。
- 高内聚、低耦合：相关逻辑放在一起，避免跨模块互相依赖过深。
- 单一职责：一个组件、函数或模块只解决一个明确问题。
- 简洁高效：优先选择直接、可读、容易维护的实现。
- 可读性优先：命名应准确表达意图，控制流程应清楚，避免炫技式写法。
- 不过度设计：只有在真实重复、真实复杂度或明确扩展需求出现时，才引入抽象。
- 尊重现有结构：改动前先阅读附近代码，沿用项目已有模式、命名和目录组织。

## 代码约束

- 不写兜底代码：不要为了“看起来更稳”而添加没有需求依据的默认值、兼容分支或吞错逻辑。
- 不写冗余代码：删除无用变量、无用分支、无用类型、无用封装和重复逻辑。
- 不写猜测性代码：不根据臆测补接口字段、数据结构、状态枚举或业务规则。
- 不隐藏问题：遇到未知响应、未知字段、未知业务规则时，应暂停并询问，或先通过项目内已有代码和数据源确认。
- 不为了保证代码绝对能运行而堆砌无意义逻辑：例如不清楚接口响应的 `id` 字段叫什么时，不得写 `id || _id || uuid || key || slug` 这类猜测链。
- 不吞异常：除非有明确业务要求，否则不要用空 `catch`、泛化提示或静默失败掩盖真实错误。
- 不引入无必要依赖：能用现有框架、标准库或项目内工具解决的，不新增依赖。
- 不做无关重构：只修改完成当前任务所必需的范围。

## 不确定时的处理方式

- 先查项目内已有实现、类型、数据文件、配置和调用点。
- 如果仍不确定，应直接询问，而不是猜。
- 如果必须临时假设，应在实现前说明假设，并让代码保持最小化。
- 对外部接口、内容结构、鉴权逻辑、数据持久化规则等不明确事项，必须确认后再写代码。

## 组件与模块规范

- 页面负责组织数据和布局，复杂展示逻辑下沉到组件。
- 组件应优先通过明确的 props 接收数据，不从无关模块隐式读取状态。
- 共享逻辑放入 `src/lib` 或已有合适模块，避免复制粘贴。
- 内容展示组件应保持纯粹，避免混入数据读取、鉴权、文件系统访问等逻辑。
- 表单、交互和服务端动作应明确输入输出，不依赖隐式副作用。

## TypeScript 规范

- 优先使用明确类型，避免滥用 `any`。
- 类型应来自真实数据结构或明确业务约定，不为猜测字段编造宽泛类型。
- 不通过类型断言掩盖不确定性，除非已经确认运行时结构。
- 校验逻辑应服务于真实边界，例如用户输入、文件内容或外部数据，而不是为了制造“万无一失”的复杂包装。

## 内容与数据规范

- 内容文件应保持结构一致，frontmatter 字段必须有明确用途。
- 新增内容类型前，应确认它与现有 notes、logs、tools、links、guestbook 等结构的关系。
- 对 slug、语言、私密内容等规则的修改要谨慎，避免破坏现有路由和内容读取。

## 验证要求

- 修改 TypeScript、React 或数据读取逻辑后，优先运行 `npm run typecheck`。
- 修改 lint 相关代码或可能触发规则的问题后，运行 `npm run lint`。
- 修改构建、路由、内容解析或 Next.js 配置后，运行 `npm run build`。
- 如果无法运行验证命令，应在交付时说明原因。
